Intelligent Job Management

v0.1.0 Prototype

GPU cluster job scheduler with automatic hardware profiling,
stoppable/resumable execution, and checkpoint-based recovery.

 ·  Jan Wangrat  · 

Architecture

API + JobRunner :8000

  • FastAPI + psycopg3 connection pool
  • Job CRUD + state machine
  • Profiling scheduler
  • Runs containers via Executor interface

Executor Abstraction

  • DockerExecutor (prototype)
  • MockSlurmExecutor (testing)
  • SlurmExecutor (Polimi cluster)

Frontend :5173

  • React 19 + TanStack Query (SPA)
  • Dashboard, Queue, Submit
  • Cluster Status, Profiling

PostgreSQL

  • jobs — metadata + scheduling decisions
  • profiling_results — per-config durations (keyed by job type)

ANDREAS-compatible API

  • job_id — reusable type identifier for profiling correlation
  • dockerImage, scriptPath, directoryToMount
  • epochsTotal, batchSize, profilingEpochsNo

Async Processes & Communication

Single-process, single-threaded asyncio — concurrency via coroutines, not threads. No message bus needed.

HTTP Layer (FastAPI)

  • Job CRUD — create, stop, resume, list, delete
  • Profiling scheduler — type-based correlation
  • schedule_lock serialises scheduling decisions
  • _queue_watcher — periodic retry for unscheduled jobs

JobRunner

  • asyncio.Queue dispatch loop
  • Concurrent container execution via Executor
  • Progress parsing (Epoch X/Y)
  • Profiling result callback → re-schedule
  • Kill-based preemption (no SIGTERM)

Request flow

HTTP POST /jobs  →  INSERT + schedule_job()  →  enqueue(instance_id)
JobRunner dequeues  →  Executor.run()  →  stream logs, parse progress
Container exits  →  on_profiling_complete() or on_job_completed()  →  re-schedule waiting jobs

Profiling Scheduler

Goal: find the fastest GPU configuration per job through incremental measurement. Each config is a JSON map like {"A40": 2}.

Exploration

  • Enumerate configs as cartesian product of profiling-node GPU counts
  • Intersect with configs production nodes can run
  • Sort by total GPU count ascending
  • Pick next un-profiled config → run on profiling node
  • Record (gpu_config, duration_seconds)

Exploitation

  • All configs profiled → pick first with available production node
  • Duration recorded but not yet used for config ranking
  • Warmup-aware estimation: drop first inter-step interval, average the rest

Profiling execution

EPOCHS_TOTAL = profiling_epochs_no   (default 100)
LOG_INTERVAL = per-job setting    (default 50 → 2 intervals, configurable lower)
Checkpoint dir: data/checkpoints/{id}/.profiling/  (isolated, cleared each run)

Worker reports result via jobs.profiling_complete → API re-queues job. When a job completes (jobs.completed), the API immediately tries to schedule waiting jobs; 60 s watcher as safety net.
